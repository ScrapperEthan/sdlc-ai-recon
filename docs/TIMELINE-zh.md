> **状态图例：** 🟢 已完成/已上线 · 🟡 进行中 · ⚪ 未开始 · 🔵 待定（选做）

---

## 一、总体目标（一句话）

给 HASE 这套**约 390 个仓库**的系统，做一个跑在**内网 GPT-5.5** 上的 **AI 工程助手**，
帮工程师**看懂系统、按公司规范改代码和建新服务**——全程基于我们自己的代码，
**只读生产、数据不出网、每条结论可核对出处**。

**更远的北极星**：把 AI 辅助覆盖整个研发流程——
需求分析 → 架构设计 → 代码生成 → 跑测试 → 构建 → 部署（部署始终人工把关）。

**为什么这件事能成 / 值钱在哪：**
这套系统本质是"**一个产品被拆成了约 390 个高度相似的仓库**"。
真正的瓶颈**不是大模型，而是"检索"**——在几百个仓库里搞清楚"改一处会牵动哪里""新服务该长什么样"。
我们投入建设的正是这层**对代码库的检索/理解能力（"护城河"）**。
**模型和界面随时可以换，这层沉淀下来的能力才是资产。**

## 三、已经做完的（阶段 0–3）

### 阶段 0 · 摸底侦察 🟢
**做了什么**：先把 390 个仓库整体扫一遍，搞清楚它到底是什么结构。
**结论**：它不是 390 个各自为政的项目，而是"**一个产品被拆成 390 个高度相似的仓库**"——
都继承同一个父模板、用同一套规范；服务之间主要靠"**消息**"通信（不是互相直接调用）。
**为什么重要**：正因为高度统一，AI 辅助才做得起来——学会一个套路，就能套到几百个仓库。

### 阶段 1 · 看懂系统 / 跨仓库问答 🟢（已上线）
**做了什么**：做出一个 AI 助手，你用大白话问它系统的问题——
比如"改了这个字段会影响哪些服务？""这条消息最后发到哪个队列？"——
它**跨仓库**给答案，而且**每条结论都附"仓库/文件:行号"的出处**，能点开核对。
**（给你 / AI 视角）** 本质就是给这套代码库做了 **RAG + agent 工具循环**：
模型不背代码，它调用我们建好的检索工具去查预建索引（代码副本、依赖图、消息路由图、调用图），
再带引用回答。**这套检索层就是护城河——模型和界面随时能换，这层换不掉。**

### 阶段 2 · 生成新服务骨架 / 脚手架 🟢（试点完成）
**做了什么**：一条命令，生成一个**符合公司统一规范**的新服务空架子
（继承统一父模板、正确的包名、目录结构、平台配置文件）。
**关键安全点**：从参考服务带出来的账号、团队、Sonar 分支策略、内部 URL 等**敏感治理值**，
会被**自动打码成 `<REVIEW>`**，不会被新服务错误继承——既是合规，也是正确性。
产物**只写进草稿目录，绝不碰生产仓库**。
**意义**：新人再也不会漏掉公司规范；这是"**代码生成**"能力的第一块落地。

### 阶段 3 · 竖切闭环 / 真实改动 🟢（薄片已跑通）
**做了什么**：不只是生成空架子，而是对一个**真实存在的服务**，生成一处**真实代码改动**（比如加一个接口），
然后**自动编译 + 跑测试到"绿"**，最后输出一个 **diff（改了哪些文件）交人评审**。全程在草稿区，**生产一行都不动**。
**现状**：这个"**看懂 → 生成真实改动 → 编译测试 → 可评审 diff**"的闭环，已经在一个真实 HASE 服务上端到端跑通。
**目前的坦白**：改动内容还是"模板化"的（比如固定加一个 `/status` 接口），
还没用检索能力去"**决定**"该改什么——那正是阶段 4 要补的。
**意义**：这是最能证明"助手能写出**可被证明可用**的改动"的一步，还一次把"**测试 + 构建**"两个环节接进了流程。

## 四、正在做 / 下一步（阶段 4）

### 阶段 4 · 让改动"由需求驱动" + 拓宽改动类型 🟡（进行中，预估 ~3–5 周）
**做什么**：给上面的闭环装上"**前门**"。
现在是人手工指定"加个 `/status` 接口"；目标是——
> 你用**一句自然语言**说需求 → 助手用检索能力**自己定位**该改哪个仓库、哪个文件 →
> 生成一份**带出处的"改动方案（spec）"** → 再据此生成代码。

同时把能改的类型从"加接口"**拓宽**到消息监听器、数据访问层、配置等真实常见改动。
**现状**：意图解析 + 定位模块已经建好（会**拒绝乱猜**，给出可核对的理由），正等在真实镜像上验证。
**为什么是现在最该做的**：它把"看得懂"和"改得动"真正接起来，也第一次让"**需求分析**"这个环节开始落地。

## 五、还要做的阶段（阶段 5–8）

### 阶段 5 · 扩到全部 390 个仓库 ⚪（预估 ~1–2 个月，可并行）
**做什么**：现在只在 **15 个仓库**的试点上跑。要覆盖全部 390 个，必须把索引**按业务域切成一块块**
（一个调用图装不下这么多），再给每个仓库配**一句话摘要**，让助手"**先缩小范围再查**"。
还要做"**索引会过期 → 定期刷新**"的机制。
**（AI 视角）** 这就是检索护城河的 **scale** 工作：分片索引 + narrow-first 路由 + 刷新流水线。
**特点**：这部分主要是**运维 / 算力活**，可以和别的阶段**并行**推进。

### 阶段 6 · 变成团队能用的服务 ⚪（预估 ~1–1.5 个月）
**做什么**：现在助手是**单机、本地、没有登录**的。要给一批同事用，得部署成**团队服务**，
加上单点登录、多用户、操作审计日志、索引刷新。
**依赖**：需要 IT / 合规提供环境和批准，**节奏不完全由我们控制**。

### 阶段 7 · 批量维护（终局能力）⚪（预估 ~2–3 个月）
**做什么**：这是最终**最值钱**的能力——一次性对**很多仓库**做统一改动，
比如统一升级某个依赖、**批量修一个安全漏洞（CVE）**。
前面所有能力（看懂 + 生成 + 编译测试）在这里被**放大到几百个仓库规模**。
**依赖**：要等前面的编译测试闭环（阶段 3/4）和规模化（阶段 5）稳定后才能上。

### 阶段 8 · 部署辅助 🔵（选做，待定）
**做什么**：银行里"上线部署"必须**人来点、层层审批**，我们**不追求 AI 自主部署**。
最多做成一个"**助手**"，帮人准备上线前检查清单、配置 diff、发布说明——**人来审核确认**。
**现状**：故意先放着，等前面闭环稳了再看。

## 六、关于预估的说明（诚实起见）

- **过去的阶段（0–3）走得很快**（以天计），是因为走的是"**Claude 写规范 → Codex 生成 → 内网验证**"的高效流水线。
- **未来阶段的时长是初步估算**，前提是继续这套交付方式、由一个人主导 + Codex 协作。
- **阶段 5、6 有外部依赖**（IT 给环境、算力、合规审批），这类依赖一旦卡住，时间会顺延——这不是技术问题。
- **"完成"要按诚实标准算**：能真正跑起来才算 🟢，"看起来像能编译"不算。
- 以上数字都可以调——这是**初稿**，我们一起对着改。
