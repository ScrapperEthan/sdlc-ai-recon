# 领导演示脚本 —— HASE SDLC 智能助手（5 分钟)

> 目标:5 分钟讲清"我们把整个 MDC 通知系统变成了一个可查询、带出处的知识库"。全程在盒子上、真数据、可点击。

## 0. 开场准备(演示前 15 分钟做一次)

```
git pull                                   # 拿到最新（含消息地图折叠）
python refresh.py --mirror "C:\Users\45509915\Downloads\HASE_MDC"   # 一条命令重建全量（~15 分钟）
```
然后起两个服务(各开一个终端):
```
set RETRIEVAL_PORT=8848 && python retrieval_service.py      # 三张视图 + 数据接口
python -m webapp.server                                     # 问答应用（可选，第 4 幕）
```
浏览器开三个标签:`http://127.0.0.1:8848/coverage.html` · `/arch.html` · `/impact.html`。

> **兜底**:就算来不及重跑 refresh,当前盒子上的数据也**完全可演示**;重跑只是让 coverage 多出"经消息连接"那一档、"其他"更小。

## 一句话定位(电梯陈述,开场先说)

> "我们把整个 MDC 通知系统 —— **392 个仓库** —— 变成了一个可查询、**每条结论都带源码出处**的知识库。它能秒答三个以前要问遍全组的问题:**改这里会影响谁、某供应商挂了哪些用例受影响、谁调用了这个服务**。覆盖**依赖、渠道、消息、函数调用**四个维度。"

---

## 第 1 幕 · 规模(仓库全景)—— `coverage.html`

- **说**:"以前这 392 个仓库对新人和救火是个黑盒。现在每一个都归位了、一个不藏。"
- **做**:指顶部 —— **392 全部归位**;各渠道泳道(SMS ~70、Email、Push…);顶部搜索框打 `tracking` 或 `sms`,**实时筛选**。指一个**空心环点** → "这个仓库自己名字不带渠道,是我们**从它的 Kafka/JMS 配置里**推出它连着 SMS 的。"
- **证明**:整个 estate 被完整建模,连"只走消息"的仓库都点亮了。

## 第 2 幕 · 流程(架构图)—— `arch.html`

- **说**:"这是一条通知怎么流的 —— 从接入、决策、到各渠道投递、出站、供应商。**每个方块都指向真实仓库。**"
- **做**:指数字带(**392 / 最大投递渠道 SMS 63 repos**);点 **SMS 投递任务** → 侧栏列出 63 个真实仓库;点 **Sinch → 查故障影响**。
- **证明**:文档里的架构图第一次和真实代码对上了。

## 第 3 幕 · 价值(故障影响)—— `impact.html`（**高潮,重点讲**)

- **说**:"救火时最关键的一问:**Sinch 这个短信供应商挂了,到底影响什么?**"
- **做**:选"故障影响" → 类型选**供应商**、值填 `sinch` → 分析。展示:**受影响仓库 ~96 个**、按关系分类(拥有渠道/投递任务/下游依赖…)、**受影响用例**,每条都能展开看**出处**。再演示"分析影响":填 `mc-hk-hase-api-tracking-core` → **下游 345 个仓库**。
- **证明**:这就是旗舰"影响通知"能力 —— 秒级、带引用、覆盖全估。

## 第 4 幕(可选)· 深度(问答应用)—— `:8765`

- **说**:"再往深一层 —— 直接问它跨仓库的调用关系。"
- **做**:问 **"谁调用了 IngressService?跨 repo 的调用链是什么?"** → 展示它给出**真实跨仓库调用者 + 源码出处**(而非关键字搜索)。
- **⚠️ 风险提示**:这是 LLM 实时回答,演示前**务必自己先跑一遍**确认它调用了 `unified_impact`、给出真实 callers。若不稳,**跳过这一幕**,前三幕已足够。
