# HASE AI 助手 —— 方案与实施规划

> 给领导汇报用。**目标：拿到批准，开始按已定的 31 个域（bundle）建 CodeGraph 索引。**
> 每部分都有专业与大白话两版。最后更新：2026-07-08

---

## 📌 一页纸看懂（先看这个）

- **我们在建什么**：给 HASE 这套约 **390 个仓库**的系统，建一个"**能看懂整套代码的 AI 助手 + 一层跨仓库理解能力（护城河）**"。
- **要解决的核心痛点**：一个 use-case / 一处代码改动进来，能**立刻知道它走哪条渠道链路（PN/SMS/WhatsApp…）、会影响哪些上下游服务，而且每条结论有出处**——这是 coding 同事的日常痛点，也是上下游系统出问题时快速响应的关键。
- **现在到哪了**：跨仓问答已上线；跨仓检索服务已建好并验证通过；390 仓已**自动分成 31 个域**、100% 覆盖、尺寸都合适。
- **要领导批的**：**开始按这 31 个域，把全量代码克隆下来、逐个域建 CodeGraph 索引**（需要存储/算力/一台常驻服务器）。批了我们就能把上面的能力从 15 仓试点推到全量 390 仓。

## 1. 我们要解决的痛点

| # | 痛点（大白话） | 谁在痛 |
|---|---|---|
| P1 | 390 个仓库，没人能说清"改一个地方会牵动哪里" | 所有开发 / 架构 |
| P2 | **一个 use-case 来了，不知道它走哪条渠道、影响哪些服务** | Coding 同事（日常）|
| P3 | **上游系统出问题，我们要快速判断哪些自己的服务受影响、准备 fallback** | 值班 / 应急 |
| P4 | **我们改了代码，要通知哪些下游 repo 会受影响**（含跨部门上下游）| 开发 / 变更管理 |
| P5 | 仓库名一堆缩写（svc、rt、hc、mdc…），检索和理解都低效 | AI 助手 / 新人 |
| P6 | 每个新需求从需求→设计→编码→测试，缺一套 AI 辅助流程 | 整个研发 |

> P2/P3/P4 其实是**同一个能力**：双向的跨仓影响分析 + 消息链路 + 出处。这是本规划的重中之重。

## 2. 现状：已经建好了什么（打勾）

- ✅ **跨仓问答 AI 上线**：大白话提问，跨仓库回答，每条带"仓库/文件:行号"出处。
- ✅ **能生成合规新服务**，并能**给真实服务改一处代码 → 自动编译测试到绿 → 出 diff**。

- ✅ **摸清友邻团队的 AI-SDLC 工作流**，定下"复用他们前端 + 插我们的跨仓理解"的互补方案。

## 3. 方案架构（一张图）

**大白话**：底层是一层"看懂代码"的能力（护城河）；中间包一个"检索服务"统一对外；上层长出三个用户能直接用的出口。**建一次，多处复用。**

```
              ┌──────────────────────────────────────────────────┐
  ~390 仓库    │  护城河：对代码库的理解层（不可替代的沉淀资产）        │
  (HASE)  ──►  │  依赖图 · 消息路由图 · CodeGraph调用图 · 术语词典/标签  │
              └───────────────────────────┬──────────────────────┘
                                          │ 只读
                              ┌───────────▼───────────┐
                              │    中心检索服务 (HTTP)   │  ← 建一次，多处复用
                              └────┬─────────┬────────┬┘
                     ┌─────────────┘         │        └─────────────┐
              ┌──────▼──────┐      ┌─────────▼────────┐    ┌─────────▼─────────┐
              │  问答 App    │      │ cross-repo-impact │    │  影响报告          │
              │ 工程师提问   │      │ skill (插进Copilot)│    │ use-case/改动 →    │
              │             │      │                   │    │ 上下游+链路+出处   │
              └─────────────┘      └───────────────────┘    └───────────────────┘
                (P1)                    (P6 互补)                (P2/P3/P4 核心)
```

**专业**：模型和界面可随时替换，唯一沉淀的资产是**对代码库的检索/理解层**。检索服务把这层能力标准化对外（只读、无副作用），因此问答 App、Copilot skill、影响报告三个上层应用共享同一套底层，不重复造。

## 4. 每个痛点怎么被解决

| 痛点 | 我们的能力 | 状态 |
|---|---|---|
| P1 改动影响面 | 依赖图 + `impact`（谁依赖我 / 我依赖谁）| 🟢 已有（全量依赖图已就绪）|
| **P2 use-case → 渠道链路** | `trace`/`usecase` 顺路由查到 topic 和渠道 | 🟡 能力具备，做成"影响报告"中 |
| **P3 上游出问题→谁受影响** | `impact` 上游方向 + `trace` | 🟡 同上 |
| **P4 改动→通知下游** | `impact` 下游方向 + 消息 `consumers/producers` | 🟡 同上 |
| P5 缩写看不懂、检索低效 | 术语词典 + 每仓业务标签（渠道/批实时/系统）→ narrow-first | 🟡 建设中 |
| P6 全流程 AI 辅助 | 复用友邻团队前端 skill + 我们的 cross-repo-impact + 深流水线 | 🟡 分阶段 |

> **关键说明（P2/P3/P4）**：一个"影响报告"功能就同时解决三个——输入 repo / use-case / topic，输出**上游依赖 + 下游受影响 + 消息链路 + 走哪条渠道，逐条带出处**。这正是 coding 同事最需要、也最能体现价值的能力。

## 5. 实施计划（分阶段，可并行）

| 阶段 | 做什么（大白话）| 需要资源 | 预估 | 依赖 |
|---|---|---|---|---|
| **A｜影响能力上线** | 做"影响报告" + 术语词典/标签，先在现有 15 仓上跑通、可演示 | 无（纯软件）| ~1–2 周 | 无（依赖图已全量）|
| **B｜建全量 CodeGraph** ⬅**要批的** | 按 31 个域，增量克隆全量代码 + 逐域建 CodeGraph 索引，接进检索服务 | **存储 + 算力 + 一台常驻服务器** | ~1–2 月（并行）| 领导批准 |
| **C｜给团队用** | fork 友邻团队前端 skill + 部署问答 App，让开发团队用起来 | 部署环境 | ~1–1.5 月 | A/B 部分就绪 |
| **D｜深流水线** | 需求→带出处的设计→真实改动→编译测试（我们已有雏形）| 无 | 后续 | C 稳定后 |

**为什么阶段 A 不用等**：估算"改动影响面/渠道链路"靠的是**依赖图和消息图，它们已经是全量的**，不需要等全量代码克隆。所以**影响能力这周就能开工、很快能演示**；建 CodeGraph（阶段 B）是给它补上"函数级调用"的深度，并行推进。

## 6. 需要领导批准 / 支持的（明确的 ask）

1. **认可方向**：以"跨仓理解（护城河）"为核心资产，服务 P2/P3/P4 这类影响分析痛点。
2. **批准阶段 B 的资源**，以便开始按 31 个域建 CodeGraph：
   - 把全量 ~390 仓的代码**克隆到内网索引机**（只读用途）；
   - **算力/存储**：31 个索引合计约 10–12 GB，构建需要一台够用的机器；
   - **一台常驻服务器**跑中心检索服务，供团队和工具调用（PoC 阶段内网直连、暂不加登录）。
3. **（可选）跨部门数据**：若要覆盖其他部门的上下游系统，需要对方提供其依赖/消息关系数据。

---

## 7. 诚实的边界与风险（提前讲清楚）

- **use-case → 渠道**的路由**不在代码里，在一张数据库表**里，我们目前只有**开发/测试环境的只读快照**。所以能查、能给出处，但**准确度取决于快照新鲜度**；要生产级准确，需定期**只读导出**生产那张表（人工把关，AI 不碰生产库）。
- **跨部门/外部系统**：只要在我们索引里的都能查；真正外部的系统需要对方共享数据，否则只能到我们估算的边界为止。
- **安全姿态不变**：只读生产、数据不出网、每条结论可核对出处、生成物只进草稿区。
