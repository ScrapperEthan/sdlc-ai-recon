# HASE SDLC 智能助手 — 现状一页纸（2026-07-15）

## 一句话
我们已经把 **整个 MDC 通知系统（392 个仓库）** 读进了一套可查询、带引用的知识层，AI 能回答"**改这里会影响谁 / 谁调用了它 / 这个供应商挂了哪些用例受影响**"——从**依赖**、**渠道**到**函数调用**三个维度，全部覆盖满量数据。

## 已建成（两层，均在满量 392 仓库上跑通）

| 能力层 | 状态 | 能回答什么 |
|---|---|---|
| **依赖 / 渠道层** | ✅ 全量 392 仓库 | 某仓库的上下游依赖；它归属/波及哪些渠道；某渠道或供应商故障 → 受影响用例与仓库（带关系分类、逐条可追溯引用） |
| **函数调用层** | ✅ 全量 31 个域包 | 跨仓库"谁调用了某个类/服务"、调用链（真实调用图，非关键字搜索） |
| **工程化** | ✅ | 一条命令全量重建知识层（`refresh.py`）+ 一条命令全量构建调用图；可重复、可增量 |

## 关键数字（演示锚点）
- **392** 个系统仓库全部打标（名称派生标签 + 业务元数据 + 依赖爆炸半径）
- **31 / 31** 个域包 CodeGraph 索引构建成功（13.3 小时，9 GiB，0 失败）
- 示例：核心库 `tracking-core` **下游 345** 个仓库；**SMS 渠道故障 → 101** 个受影响仓库；`IngressService` 有 **13** 个真实跨仓库调用者
- 架构图：42 个管线节点，13 个映射到真实仓库（其余为外部系统/供应商/Kafka 主题，按设计留白）

## 还差什么（到"影响通知"终极目标）
- **消息地图（Message Map）**：目前渠道波及靠 Maven 依赖，覆盖 163/392；其余 ~214 个仓库通过 **Kafka 消息**连接渠道，需要抽取 publish/subscribe 图才能补全。**这是下一个核心工程。**
- 小尾巴：架构图上 HARO / CN-Gateway 两个纯基础设施节点的人工绑定；构建稳定后用全量图重生域包划分。

## 演示怎么讲（三步）
1. **看全景** — 架构图：整个通知管线，每个节点标注背后真实仓库数（SMS 最大 63）。
2. **点进去** — 点任意节点 → 它的真实仓库 + 一键"查故障影响"（某渠道/供应商挂了 → 受影响用例）。
3. **问 AI** — 问答应用："谁调用了 IngressService？" → AI 给出跨仓库真实调用链 + 出处引用。
