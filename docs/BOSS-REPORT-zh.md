# HASE AI 工程助手 —— 阶段汇报

## 一句话
我们做了一个跑在**内网 GPT-5.5** 上的 AI 工程助手，能帮工程师**看懂**这套约 390 个仓库的系统、并**按公司规范生成**新服务——全程基于我们自己的代码、**只读生产、数据不出网**。

## 为什么值得做
这套系统本质是**一个产品被拆成了约 390 个高度相似的仓库**。真正的瓶颈不是大模型，而是**"检索"**——在 390 个仓库里搞清楚：一个改动会牵动哪些地方、一个新服务该长什么样。我们建的正是这一层能力（"护城河"）。**模型和界面都可以随时替换，这层对代码库的检索/理解能力才是沉淀下来的资产。**

## 目前已经能用的

**① 理解（已上线）**
用大白话提问，例如："改了 `EventPayload` 会影响哪些服务？""ingress 的消息是怎么路由到某个 topic 的？"
助手会**跨仓库**给出影响面，并且**每一条结论都附带"仓库/文件:行号"的出处**可核对。已在一个 15 仓库的真实业务流（ingress→消息→追踪）上端到端跑通。

**② 生成（试点，刚完成）**
一条命令 → 生成一个**符合公司统一规范**的新服务骨架（继承统一父 POM 与 starter、Java 21、包名规范、平台配置、目录结构）。
关键：参考服务里的**账号、团队、Sonar 分支策略、内部 URL、组织业务码**等，都会被**自动打码成 `<REVIEW>`**，不会被新服务错误继承——既是合规，也是正确性。产物**只写进草稿目录，绝不碰生产仓库**。

## 3 分钟现场演示（建议这样讲）
1. **理解** —— 问一个影响分析问题 → 跨仓库、带出处的回答。*"它读的是我们自己的代码，不是通用知识。"*
2. **生成** —— 一条命令生成一个合规新服务，几秒钟。*"新人也不会漏掉公司规范。"*
3. **安全** —— 指出：只写进草稿区、每条有据可查、敏感治理值自动打码。*"可管控、可审计、碰不到生产。"*

## 安全合规（面向银行的姿态）
只读生产 · 跑内网 GPT-5.5、**数据不出网** · 每条结论**可核对出处** · 生成物**只在草稿区**且治理值**自动打码** · 纯标准库、适配隔离网环境。

## 在整个研发流程里的位置 + 下一步
| 阶段 | 状态 |
|---|---|
| 需求分析 | 未开始 |
| **架构理解 / 影响分析** | 🟢 **已上线** |
| **代码生成** | 🟡 **起步（脚手架已完成试点）** |
| 测试 / 构建 | ⚪ 尚未 |
| 部署 | 银行内保持**人工把关**（不追求 AI 自主） |

**下一步（建议投入）**：打通一个**"竖切"闭环**——让助手不只生成空骨架，而是**给现有服务生成一处真实代码改动 → 自动编译 + 跑测试到"绿" → 输出 diff 交人评审**。这一步把骨架推进到"**能被证明可用的改动**"，同时把"测试"和"构建"两个环节接进流程。这是最能让人信服、也最能对上"辅助日常研发"目标的下一个能力。

## 需要的支持（按需，可删改）
- 认可这个方向、继续投入下一步"竖切闭环"；
