# 领导演示脚本 —— HASE SDLC 智能助手（5 分钟)

> 目标:5 分钟讲清"我们把整个 MDC 通知系统变成了一个可查询、带出处、能实时作答的知识库"。全程在盒子上、真数据、可点击。

## 0. 开场准备(演示前 15 分钟做一次)

```
git pull
python refresh.py --mirror "%SDLC_MIRROR%"     # 一条命令重建全量（~15 分钟）
```
起两个服务(各开一个终端);模型/镜像/流式都已是用户环境变量,直接起即可:
```
set RETRIEVAL_PORT=8848 && python retrieval_service.py     # 三张视图 + 数据接口
python -m webapp.server                                    # 问答应用（第 4 幕，LLM_STREAM=1 已在环境变量里）
```
浏览器开四个标签:`/coverage.html` · `/arch.html` · `/impact.html`(都在 :8848),外加问答页 `http://127.0.0.1:8765`。

> **兜底**:就算来不及重跑 refresh,盒子上的数据也**完全可演示**。
> **演示前务必自己把第 4 幕那句问题跑一遍**(见下),确认逐字流式 + 出流程图。

## 一句话定位(电梯陈述,开场先说)

> "我们把整个 MDC 通知系统 —— **392 个仓库** —— 变成了一个可查询、**每条结论都带源码出处**、还能**逐字实时作答**的知识库。它秒答三个以前要问遍全组的问题:**改这里会影响谁、某供应商挂了哪些受影响、谁调用了这个服务**。覆盖**依赖、渠道、消息、函数调用**四个维度。"

---

## 第 1 幕 · 规模(仓库全景)—— `coverage.html`

- **说**:"以前这 392 个仓库对新人和救火是个黑盒。现在每一个都归位了、一个不藏。"
- **做**:指顶部 **392 全部归位**;各渠道泳道(SMS、Email、Push…);顶部搜索框打 `tracking` 或 `sms` **实时筛选**。指一个**空心环点** → "这个仓库自己名字不带渠道,是我们**从它的 Kafka/JMS 配置**里推出它连着 SMS 的。"
- **证明**:整个 estate 被完整建模,连"只走消息"的仓库都点亮了。

## 第 2 幕 · 流程(架构图)—— `arch.html`

- **说**:"这是一条通知怎么流的 —— 从接入、决策、到各渠道投递、出站、供应商。**每个方块都指向真实仓库**,布局干净不重叠。"
- **做**:指数字带(**392 / 最大投递渠道 SMS 63 repos**);点 **SMS 投递任务** → 侧栏列出 63 个真实仓库。
- **证明**:文档里的架构图第一次和真实代码对上了。

## 第 3 幕 · 价值(故障影响)—— 高潮,重点讲(在 `arch.html` 或 `impact.html`)

**这一幕的杀手锏:我们不喊狼来了。**

- **说**:"救火最关键一问:**Sinch 这个短信供应商挂了,到底影响什么?**"
- **做(供应商级)**:点 **Sinch 供应商节点 → 查故障影响** → **约 24 个受影响仓库**(投递任务 / 出站 API / 下游依赖)。
- **对比(渠道级)**:点 **SMS 投递任务 → 查故障影响** → **一百多个**(整条短信渠道)。
- **说这句(credibility 关键)**:"注意区别:**Sinch 一家挂了是 24 个,不是把整条短信渠道一百多个都算上** —— CSL、3HK 还在送。想看整条渠道的最坏情况?一键就有。**我们只报真受影响的,不夸大。**"
- **再加一枪(依赖影响)**:去 `impact.html`,分析 `mc-hk-hase-api-tracking-core` → **下游数百个仓库**,每条可展开看出处。
- **证明**:这就是旗舰"影响通知"能力 —— 秒级、带引用、**供应商级和渠道级分得清**。

## 第 4 幕 · 深度(问答应用,逐字流式)—— `:8765`（**现在是稳定亮点,别跳**）

- **说**:"最深一层 —— 用大白话问它跨仓库的调用关系,它**带着源码、逐字实时**答给你。"
- **做**:把下面这句**原样粘进去**(详细问法每次都出流程图,实测 5/5 稳):

  > 谁调用了 IngressService？跨 repo 的调用链是什么？列出真实的跨仓库调用者，并给出每个调用者的源码 file:line。

- **看点**:① 答案**逐字打出来**(真流式,不是转圈等 40 秒);② 出一张 **Mermaid 流程图**,多个跨仓库调用者(含 `mc-hk-hase-api-campaign-core` 的 `SendCampaignEventService`)收敛到 `IngressService.publishIngressEvent`;③ 下方 **Evidence 每条都带 `仓库/文件:行号`**、引用校验全绿。
- **证明**:不是关键字搜索 —— 是真实的跨仓库函数调用图,精确到行,带出处,还能实时看它生成。

## 收尾 & 应答

**收尾一句**:"我们从'估算是个黑盒'做到了'整个系统可查询、带出处、四个维度全覆盖,还能实时作答'。"

**如果被问"下一步"**:
- 让"完整调用者清单"确定性化(检索侧固定注入,永不漏);
- 接入变更评审工作流(改动 → 自动生成影响+通知清单)。

**如果被问"数据准不准"**:每条结论都带 `仓库/文件:行号` 或图/表出处,可当场展开核对 —— 这是刻意设计:**不编造,存疑就标"待核实",供应商级不夸大成渠道级**。

**演示口令备忘**:第 4 幕那句问题、第 3 幕"Sinch 24 vs 渠道一百多"的对比 —— 这两处最出彩,练熟。
